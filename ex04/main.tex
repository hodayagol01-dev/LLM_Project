% 3. Why LaTeX is used for reproducible PDF generation
\section{The Role of LaTeX in Reproducible PDF Generation}
LaTeX remains a cornerstone in academic publishing and technical documentation due to its unparalleled capabilities in producing high-quality, professional-looking documents. Its strength lies not just in typesetting sophisticated mathematical formulas and complex layouts, but fundamentally in its approach to document structure and reproducibility. Unlike WYSIWYG editors, LaTeX separates content from presentation; authors focus on the logical structure of their document (sections, paragraphs, lists, citations), and LaTeX's engine handles the visual rendering based on predefined styles and templates. This logical separation is crucial for reproducibility. A LaTeX source file, when compiled on any system with a compatible LaTeX distribution, will produce the exact same output PDF. This consistency is vital for scientific research, where precise presentation of data and findings is paramount. Furthermore, LaTeX source files are plain text, making them ideal for version control systems like Git. This allows for tracking changes, collaborating effectively, and reverting to previous versions, all of which are essential components of a reproducible workflow.

The stability and maturity of the LaTeX ecosystem mean that documents compiled years apart can still yield identical results, provided the same distribution and packages are used. This long-term stability is a significant advantage over many modern document creation tools. The ability to define custom commands and environments, and to integrate external data sources or scripts, further enhances LaTeX's power for complex, large-scale document generation projects, making it an ideal choice for the framework developed in this assignment.

% 4. Why LuaLaTeX is useful
\section{Advantages of LuaLaTeX}
While traditional LaTeX engines like pdfLaTeX are powerful, LuaLaTeX offers significant advancements that are particularly beneficial for complex, automated document generation. LuaLaTeX integrates the Lua scripting language directly into the typesetting process. This allows for more sophisticated document manipulation and control than is possible with standard LaTeX macros alone.

Key advantages of LuaLaTeX include:
\begin{itemize}
    \item 	extbf{Embedded Scripting:} Developers can write Lua scripts that run at compile time. These scripts can dynamically generate content, modify document structures, interact with external data, or perform complex calculations before the final typesetting occurs. This capability is invaluable for automating the creation of detailed reports, bibliographies, or index entries.
    \item 	extbf{Enhanced Unicode Support:} LuaLaTeX provides superior support for Unicode characters, which is essential for documents dealing with multiple languages, specialized symbols, or modern character sets. This makes it more flexible for internationalization and diverse content needs.
    \item 	extbf{Extensibility:} The Lua API exposed to LaTeX allows for deep customization and extension of LaTeX's capabilities. Package authors can leverage Lua to create more powerful and efficient packages, and users can implement custom logic directly within their documents.
    \item 	extbf{Performance:} In many scenarios, LuaLaTeX can offer performance improvements, especially for documents that require extensive processing or dynamic content generation, due to the efficiency of the Lua interpreter.
\end{itemize}
For an assignment focused on agentic PDF generation, LuaLaTeX's ability to programmatically control document creation and structure makes it a powerful tool. It allows the 'LaTeX Builder Agent' to execute sophisticated logic, thereby enabling the generation of more dynamic and complex documents that would be challenging with older engines.

% 5. Role of the CLS template
\section{The Role of the CLS Template}
The Document Class file (``.cls``) serves as the blueprint for a LaTeX document's overall structure, appearance, and behavior. In this assignment, the 	exttt{custom-academic-template.cls} file plays a critical role in enforcing consistency, adhering to specific academic standards, and streamlining the document creation process.

Its responsibilities include:
\begin{itemize}
    \item 	extbf{Defining Document Structure:} It sets up the basic commands and environments for sections, subsections, figures, tables, bibliographies, and other standard document elements. This ensures a uniform structure across all generated documents.
    \item 	extbf{Enforcing Styling and Formatting:} The CLS file dictates the typography (fonts, sizes, line spacing), margins, page layout, and the visual presentation of headings, lists, and other elements. This guarantees a professional and aesthetically pleasing academic look, consistent with the assignment's requirements.
    \item 	extbf{Managing Packages:} It loads and configures essential LaTeX packages required for the document's functionality. By managing package inclusion here, we ensure that all necessary tools for typesetting, referencing, or specific formatting are available without cluttering the main document file.
    \item 	extbf{Providing Custom Commands and Environments:} The template can define custom commands or environments tailored to the specific needs of the project, such as commands for specific types of figures, tables, or even agent-generated content blocks.
    \item 	extbf{Ensuring Reproducibility:} By centralizing formatting and package management, the CLS file ensures that documents compiled on different systems using the same template and LaTeX distribution will look identical, contributing significantly to the overall reproducibility of the generated PDFs.
\end{itemize}
In essence, the 	exttt{custom-academic-template.cls} acts as a contract between the content generators (agents) and the typesetting engine, ensuring that the output is both structurally sound and visually compliant with academic conventions.

% 6. Skills as reusable agent instructions
\section{Skills as Reusable Agent Instructions}
In the context of agent-based systems, "skills" represent modular, self-contained units of functionality that an agent can invoke to perform specific tasks. For this assignment, skills are designed as declarative instructions or scripts that encapsulate specialized knowledge or actions required for document generation. They promote a clean separation of concerns, allowing the orchestrator agent to manage the overall workflow while delegating specific sub-tasks to agents equipped with the appropriate skills.

The benefits of using skills are manifold:
\begin{itemize}
    \item 	extbf{Modularity:} Each skill focuses on a single responsibility (e.g., writing a specific section, checking citations, compiling LaTeX). This makes the system easier to understand, develop, and maintain.
    \item 	extbf{Reusability:} A skill developed for one part of the document generation process can be reused in other contexts or by different agents, promoting efficiency and consistency. For instance, a 'QA References Skill' could be applied to various document sections.
    \item 	extbf{Abstraction:} Agents interacting with skills do not need to know the intricate details of how a skill is implemented. They only need to know what input the skill requires and what output it produces. This simplifies agent design and interaction.
    \item 	extbf{Scalability:} As the complexity of the document generation task grows, new skills can be added or existing ones can be improved without significantly altering the core orchestrator logic.
    \item 	extbf{Testability:} Individual skills can be tested in isolation, simplifying the debugging process and ensuring the reliability of each component.
\end{itemize}
The skills used in this assignment, such as 'Content Writer', 'LaTeX Builder', and various 'QA Agents', are designed to be callable by the orchestrator, forming the building blocks of the autonomous PDF generation pipeline.

% 7. Orchestrator Agent Design
\section{Orchestrator Agent Design}
The Orchestrator Agent is the central intelligence of the autonomous PDF generation system. Its primary responsibility is to manage the entire workflow, from understanding the overall goal (generating a comprehensive academic report) to breaking it down into manageable sub-tasks, delegating these tasks to specialized skills, and coordinating their execution.

The design of the orchestrator agent follows a goal-oriented, state-driven approach:
\begin{itemize}
    \item 	extbf{Goal Decomposition:} Upon receiving the request to generate the document, the orchestrator breaks down the high-level objective into a series of sequential and parallel sub-goals. This decomposition is informed by the assignment requirements, including the specific sections to be generated and the desired document structure.
    \item 	extbf{Task Delegation:} For each sub-goal, the orchestrator identifies the most appropriate skill or agent to handle it. For example, generating text for a section is delegated to the 'Content Writer Agent', while compiling the LaTeX document is assigned to the 'LaTeX Builder Agent'.
    \item 	extbf{Workflow Management:} The orchestrator manages the dependencies between tasks. For instance, content must be written before it can be compiled, and compilation must succeed before QA checks can be performed on the output PDF. It ensures tasks are executed in the correct order, potentially in parallel where dependencies allow.
    \item 	extbf{Error Handling and Recovery:} A critical function of the orchestrator is to monitor the execution of delegated tasks. If a skill fails or produces an error (e.g., LaTeX compilation errors), the orchestrator must detect this, attempt recovery strategies (e.g., re-running a task with modified parameters, using an alternative approach), or report the failure to a higher authority (e.g., the user).
    \item 	extbf{State Management:} The orchestrator maintains the current state of the document generation process, tracking which tasks have been completed, which are in progress, and which are pending. This state information is crucial for resuming interrupted processes or for providing status updates.
\end{itemize}
The orchestrator acts as the conductor of an orchestra, ensuring that each instrument (skill) plays its part at the right time, contributing to a harmonious final performance—in this case, a polished, accurate, and reproducible PDF document.

% 9. LaTeX Builder Agent
\section{LaTeX Builder Agent}
The LaTeX Builder Agent's core function is to take the content generated by other agents and assemble it into a fully compilable LaTeX document, ultimately producing the final PDF output. This agent acts as the bridge between the abstract content and the concrete, typeset document, leveraging the power of LuaLaTeX and the project's custom CLS template.

Its responsibilities are as follows:
\begin{itemize}
    \item 	extbf{Document Assembly:} It receives the structured content (e.g., text for sections, tables, figure captions) and embeds it within the LaTeX structure defined by the 	exttt{custom-academic-template.cls}. This involves placing content in the correct sections, ensuring proper use of LaTeX commands for formatting and structure.
    \item 	extbf{Compilation Execution:} The agent invokes the LuaLaTeX compiler to process the assembled `.tex` file. This step transforms the source code into a PDF. The agent must ensure that the correct compiler is used and that any necessary compilation passes (e.g., for references, table of contents) are executed.
    \item 	extbf{Error Handling:} If the LuaLaTeX compilation process fails, the agent is responsible for capturing the error messages. It then reports these errors back to the orchestrator, which may attempt to resolve them or use them to refine subsequent generation steps.
    \item 	extbf{Output Management:} Upon successful compilation, the agent ensures that the generated PDF file, along with any intermediate files (like `.aux`, `.log`), are managed appropriately, typically making them available for the QA agents.
    \item 	extbf{Configuration:} It ensures that the compilation environment is correctly configured, respecting any specific requirements of the CLS template or the LuaLaTeX engine.
\end{itemize}
This agent is critical for translating the abstract "what" of the content into the concrete "how" of the typeset document, directly contributing to the assignment's goal of automated, high-quality PDF generation.

\section{QA References Agent}
The QA References Agent is responsible for ensuring the accuracy and integrity of all citations, bibliographical entries, and cross-references within the generated document. In academic writing, correct referencing is paramount for credibility and for avoiding plagiarism. This agent automates the verification of these critical elements.

Its key tasks include:
\begin{itemize}
    \item 	extbf{Citation Verification:} The agent checks that every citation made in the text (e.g., 	exttt{\cite\{key\}}) corresponds to a valid entry in the bibliography. It verifies that citation keys are correctly spelled and formatted.
    \item 	extbf{Bibliography Integrity:} It ensures that all entries in the bibliography section are correctly formatted according to the specified style (as dictated by the CLS template or associated bibliography style files) and that all cited works are present.
    \item 	extbf{Cross-Reference Validation:} For internal references (e.g., to figures, tables, sections using 	exttt{
ef\{label\}}, 	exttt{\pageref\{label\}}), the agent confirms that the labels exist and that the referenced numbers or page numbers are accurate.
    \item 	extbf{Unused References Detection:} It can identify citation keys or labels that are defined but never used, helping to clean up the document.
\end{itemize}
By performing these checks, the QA References Agent guarantees that the document's scholarly apparatus is robust and error-free, contributing significantly to its overall quality and trustworthiness. The correct functioning of this agent relies on the LaTeX Builder Agent successfully processing the bibliography and references during compilation.

% 16. Error Handling During Compilation
\section{Error Handling During Compilation}
LaTeX compilation, especially with sophisticated engines like LuaLaTeX and custom templates, can encounter various errors. A robust error handling strategy is crucial for the success of the autonomous generation pipeline. The LaTeX Builder Agent, in coordination with the Orchestrator, is responsible for detecting, reporting, and potentially resolving these issues.

Common types of LaTeX compilation errors include:
\begin{itemize}
    \item \textbf{Syntax Errors:} Incorrect LaTeX commands, missing braces, unmatched environments, unsafe inline examples, or typos in commands.
    \item 	extbf{Undefined Control Sequences:} Using commands that are not recognized, often due to missing packages or typos.
    \item 	extbf{Missing Files:} The compiler cannot find required files, such as images, bibliography data, or custom style files.
    \item 	extbf{Package Conflicts:} Incompatible packages loaded into the document, leading to unpredictable behavior or outright failure.
    \item 	extbf{Floats Placement Errors:} Issues with figures or tables not fitting on the page or causing layout problems.
    \item 	extbf{Reference Errors:} Citations or cross-references to labels that do not exist or are not correctly processed.
    \item 	extbf{Encoding Issues:} Problems with character encoding, especially when mixing different languages or special symbols.
\end{itemize}

The error handling process typically involves:
\begin{enumerate}
    \item 	extbf{Capturing Output:} The LaTeX Builder Agent captures the standard output and error streams from the LuaLaTeX compiler. The 	exttt{.log} file generated by LaTeX is also a primary source of detailed error information.
    \item 	extbf{Error Identification:} The agent parses these outputs to identify specific error messages, often indicated by lines starting with 	exttt{! LaTeX Error:} or similar. It attempts to pinpoint the error message, the relevant line number in the `.tex` file, and the context.
    \item 	extbf{Reporting to Orchestrator:} Identified errors are reported back to the Orchestrator Agent. The Orchestrator then decides on the next course of action.
    \item 	extbf{Recovery Strategies:} Depending on the error type, the Orchestrator might:
        \begin{itemize}
            \item 	extbf{Retry Compilation:} For transient issues, a simple retry might suffice.
            \item 	extbf{Modify Content:} If an error is due to malformed content generated by the Content Writer, the Orchestrator might request a revision of that content.

            \item 	extbf{Inform User:} If an error is unrecoverable by the agents, the user is notified with detailed diagnostic information.
        \end{itemize}

By systematically handling compilation errors, the system aims to produce a reliable output and provides valuable feedback for improving the generation process.

\section{Limitations}
Despite the comprehensive nature of the agentic PDF generation framework, several limitations are inherent in the current implementation and the scope of this assignment. Recognizing these limitations is crucial for understanding the system's capabilities and for planning future development.

\begin{itemize}

The ability to recover from all possible failure modes is challenging.
    \item 	extbf{Environment Specificity:} While aiming for reproducibility, perfect replication across all environments can still be challenging. The reliance on specific tool versions (e.g., LuaLaTeX version) can be a factor.
    \item 	extbf{Real-time Interaction Limitations:} The system operates in a batch processing mode. It is not designed for real-time, interactive document editing or collaborative writing in the way a human user might experience with a word processor.
    \item 	extbf{Limited Understanding of Nuance:} Agents may struggle with highly nuanced instructions, implicit requirements, or understanding the broader context of a document in the same way a human author would.
\end{itemize}
These limitations highlight areas where further research and development are needed to achieve fully autonomous and human-level quality in document generation.

% 23. Appendix C: Compilation Troubleshooting
\section{Appendix C: Compilation Troubleshooting}
Encountering compilation errors is a common part of working with LaTeX. This section provides guidance on troubleshooting typical issues that may arise during the automated generation process.

\subsection{Common Errors and Solutions}

\subsubsection{Undefined Control Sequence / Unknown Command}
    	extbf{Symptom:} 	exttt{! Undefined control sequence.} or 	exttt{! LaTeX Error: Unknown command 	extbackslash commandname.}
    	extbf{Cause:} A command was used that is not defined, either because it's misspelled, requires a package not loaded, or is from a package not installed.
    	extbf{Solution:}
    \begin{itemize}
        \item Check for typos in the command name.
        \item Ensure the required package is loaded in 	exttt{custom-academic-template.cls}. If not, and adding packages is allowed, add 	exttt{\usepackage\{packagename\}}.
        \item Verify that the package is installed in your LaTeX distribution (e.g., via 	exttt{tlmgr install packagename} if using TeX Live, or managed by BasicTeX).
    \end{itemize}

\subsubsection{File Not Found}
    	extbf{Symptom:} 	exttt{File `filename.ext' not found.}
    	extbf{Cause:} The LaTeX compiler cannot locate a required file (e.g., an image, a bibliography file, or a style file).
    	extbf{Solution:}
    \begin{itemize}
        \item Ensure the file exists at the specified path relative to the main `.tex` file or the specified output directory.
        \item Check that the file path in the LaTeX command (e.g., 	exttt{\includegraphics\{path/to/image.png\}}) is correct.
        \item Verify that intermediate build artifacts are not preventing access if compilation is complex.
    \end{itemize}

\subsubsection{Missing Bibliography Entry / Undefined Citation}
    	extbf{Symptom:} 	exttt{LaTeX Warning: Citation `key' on page X undefined on input line Y.} or 	exttt{There were undefined citations.}
    	extbf{Cause:} A citation key used in the text does not exist in the bibliography file, or the bibliography was not processed correctly.
    	extbf{Solution:}
    \begin{itemize}
        \item Ensure the citation key used in 	exttt{\cite\{key\}} exactly matches an entry's key in 	exttt{references.bib}.
        \item Verify that the bibliography processing steps (e.g., running 	exttt{bibtex} or 	exttt{biber}, depending on the CLS setup) are executed correctly as part of the compilation workflow. Multiple 	exttt{lualatex} runs are essential.
    \end{itemize}

\subsubsection{Package Errors / Conflicts}
    	extbf{Symptom:} Errors related to package options, 	exttt{Redefinition} warnings, or general compilation failure after loading a package.
    	extbf{Cause:} Packages might have incompatible options or conflict with each other.
    	extbf{Solution:}
    \begin{itemize}
        \item Consult the documentation for the conflicting packages.
        \item Simplify package loading in 	exttt{custom-academic-template.cls}. If a package is not strictly necessary, remove it.
        \item Ensure packages are loaded in a sensible order.
    \end{itemize}

\subsubsection{Infinite Loops}
    	extbf{Symptom:} Compilation seems to hang or repeat indefinitely, often with repetitive output.
    	extbf{Cause:} Typically caused by recursive definitions, incorrect cross-referencing, or issues with packages that modify document flow.
    	extbf{Solution:}
    \begin{itemize}
        \item Examine the log file for repetitive patterns.
        \item Simplify the document by commenting out sections or disabling specific packages to isolate the cause.
        \item Check for incorrect use of 	exttt{\label} and 	exttt{
ef}.
    \end{itemize}

\subsection{Debugging Strategy}
When errors occur, the following strategy is recommended:
\begin{enumerate}
    \item 	extbf{Examine the Log File:} Always start by inspecting 	exttt{ex04/main.log} for the most detailed error messages.
    \item 	extbf{Isolate the Error:} Use 	exttt{--interaction=nonstopmode} to see all errors, or temporarily comment out sections of the 	exttt{.tex} file to determine which part triggers the error.
    \item 	extbf{Incremental Compilation:} Compile smaller parts of the document or known working examples to verify components.
    \item 	extbf{Consult Documentation:} Refer to the LaTeX Companion, package documentation, or online resources (like TeX Stack Exchange) for specific error messages.
\end{enumerate}

\section{Appendix I: Risk Review and Mitigation}

One risk in this type of project is that the final document may become disconnected from the repository structure. This risk is mitigated by describing the repository components inside the report and by maintaining README files that explain the same structure from a practical perspective. The report and the repository therefore support each other.

A second risk is compilation fragility. LaTeX workflows may depend on packages that are not installed on every machine. This project mitigates that risk by keeping the template simple and by avoiding unnecessary dependencies. When compilation issues appeared, the preferred solution was to simplify the document structure rather than introduce additional fragile requirements.

A third risk is over-reliance on automatic generation. Agentic tools can produce content quickly, but they can also introduce formatting mistakes or unsafe technical examples. The workflow therefore combines automated generation with human inspection, terminal validation, and repeated compilation attempts. This combination reflects a realistic use of AI-assisted development.

A fourth risk is unclear grading evidence. A repository can contain many files, but without documentation the evaluator may not understand their purpose. The README files, logs, skills, and QA materials are included to make the submission legible, auditable, and easier to evaluate.
